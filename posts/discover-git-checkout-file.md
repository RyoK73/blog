## はじめに

Gitを活用して開発していると、「あ〜やっぱりあそこは前の実装のほうが良かったな〜」と感じる場面に多々遭遇します。
そもそも私がGit大好きになったのも、VBAなんかのデフォでバージョン管理できない言語で開発してるときに、ロールバックしたくなることが多かったからなんです。

当時Pythonを書き始めたときに、まっさきに使い出したのがGitでした。

ということで今回はそんなファイル巻き戻しに関するお話です。

## ファイル内容を巻き戻すには

私は普段この２つを使います。

1. `git reset --hard (<hash> or HEAD@{#})`
2. `git checkout (<hash> or HEAD@{#}) --filepath`

## `git reset --hard`

私はGit触り始めた当初は、`reset --hard`ばかり使ってました。
というのも、ファイル内容を巻き戻すのって**undo**のイメージが強く、「やっぱやーめた」という気持ちが大きかったからです。
試行錯誤の履歴を見せたくないみたいな...

ただ、個人開発をしてると、むしろその試行錯誤の履歴こそ重要なドキュメントになるなと感じることが増えました。
「〇〇を検討したけど、□□という理由で使えなかったからその実装をやめた」なんていうことになるわけです。
ここで `reset --hard`をすると、最初からその設計だったのか、意図があっての設計なのかが見えなくなってしまいます。
あと、`reset --hard`をするとそこまでの履歴がさっぱり消え去ってしまうので、間違ったポイントまで戻ってしまったときに戻すのがめんどくさくなりますw

## `git checkout`

`reset --hard`との違いを端的に言えば、**それまでの変更も履歴に残ること**です。
